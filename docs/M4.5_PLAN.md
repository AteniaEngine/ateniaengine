# M4.5 Plan — Real Model Execution

## Context

M4 closed loader mechanics (commits `229fb25` to `36e4e18`). M4.5
aims at executing a real model end-to-end: TinyLlama 1.1B.

Architectural gap identified in `HANDOFF_APX_V20_M4.md`:

- RoPE (does not exist as a `NodeType`)
- TinyLlama graph builder (does not exist)
- Numerical validation against a PyTorch reference (does not
  exist)

M4.5 covers these gaps.

---

## Target confirmed

**TinyLlama 1.1B Chat** (BF16 native).

Rationale:

- Fits the 4070 Mobile dev box: 8 GB VRAM + 16 GB RAM (the
  checkpoint is ~2.2 GB BF16, ~4.4 GB after the M4-d downcast to
  F32 — exceeds VRAM, fits in RAM).
- Architecture is plain Llama 2: standard RoPE, no GQA, no
  context-extension scaling.
- Single-file safetensors (`model.safetensors`); no sharded-
  loader work needed.
- First real exercise of the BF16 → F32 decode path landed in
  M4-d. The M4 empirical validation (commit `06eb526`) used gpt2
  which is F32; TinyLlama exercises BF16 against a real file for
  the first time.
- PyTorch reference is trivially available via `transformers`:
  `AutoModelForCausalLM.from_pretrained("TinyLlama/...")`.

Llama 3.2 1B was rejected as a starting target:

- Requires GQA (8 KV heads vs 32 Q heads); buildable from
  primitives but adds reshape + broadcast machinery to M4.5-b.
- Requires "llama3" scaling RoPE (custom piecewise scaling
  function); adds an M4.5-a sub-variant that does not need to
  exist for TinyLlama.
- Larger numerical-debug surface for M4.5-d.

If TinyLlama lands and Llama 3 becomes the next target, the work
to add GQA + llama3 scaling is incremental on top of a working
M4.5 foundation, not concurrent risk.

## Sub-phases

### M4.5-a — RoPE NodeType (2 to 2.5 days)

Implement Rotary Positional Embedding as a dedicated `NodeType`.

#### Architectural decisions (locked post-investigation)

1. **Layout: half-split** (HuggingFace Transformers convention),
   NOT interleaved (original RoPE paper).

   Rationale: PyTorch reference uses half-split. Implementing
   interleaved would break numerical equivalence in M4.5-d with
   debug messages that look like correctness bugs but are layout
   bugs. The two layouts are mathematically the same rotation
   applied to different pairs of dimensions; their outputs differ
   bit-for-bit on the same input.

2. **NodeType definition**:
   `NodeType::RoPE { head_dim: usize, base_freq: u32 }`.

   `base_freq` is `u32` (not `f32`) deliberately: the pre-M4.5
   `NodeType` enum derives `PartialEq + Eq`, and adding an `f32`
   variant breaks `Eq` (Rust forbids it because of NaN). All
   known production values are exact integers (TinyLlama: 10000;
   Llama 3: 500000). Cast to `f32` inside the handler. Zero loss
   of expressiveness, derive preserved.

3. **Position handling**: implicit `[0..seq_len)` for M4.5.
   Document explicitly in the `NodeType` doc-comment that KV cache
   (M5+) will require extension to a second input carrying explicit
   position offsets. Forward signature stays single-input for now.

4. **Frequency computation**: on-the-fly per forward, no
   precomputed cos/sin tables.

   Cost: ~131,072 `f32::sin` + `f32::cos` calls per forward at
   TinyLlama context (2048 × head_dim/2 × 2). ~1.3 ms in debug.
   Negligible against the matmuls that dominate any real
   transformer forward.

   If M5+ brings context ≥32k, reconsider: cost scales linearly
   to ~20 ms per forward, at which point a `thread_local!` cached
   table becomes worthwhile. Refactor would be internal to the
   handler, no NodeType API change.

5. **Implementation pattern**: clone from `NodeType::SiLU` /
   `NodeType::Softmax` in `src/amg/graph.rs` — both are 1 input
   / 1 output element-wise ops with a math formula and a tape-
   recorded backward closure. RoPE fits the same shape.

|---|---|---|

Total: ~420 LOC of code + fixture + script.

#### Tests

1. **Forward bit-exact vs PyTorch fixture** — canary for layout
   bugs. The most important single test: catches half-split vs
   interleaved confusion immediately. Tolerance 1e-6 against a
   fixture committed to the repo (generated once with HF
   transformers' `apply_rotary_pos_emb`).
2. **Rotation properties** — position 0 = identity; L2 norm
   preserved; double rotation matches single double-angle.
3. **Backward via grad check** — analytical backward closure vs
   numerical finite-difference gradient. Tolerance ~1e-3 (finite-
   diff noise).
4. **Multi-head independence** — head 0 and head 1 produce
   different rotated values for the same input pattern.
5. **Integration in a mini-graph** — Input → Linear → RoPE →
   Output, verify forward executes without crash and shape is
   preserved.

#### Commit shape

Single commit: **"Add RoPE NodeType with half-split layout
(M4.5-a)"**.

Escape hatch: if implementation overruns 2.5 days during the
sprint, close intermediate with forward + Test 1 only; backward
+ Tests 2–5 in a follow-up commit. Forward + canary test is the
minimum viable RoPE — the graph builder in M4.5-b can proceed
with that level if needed.

### M4.5-b — TinyLlama graph builder (1 to 2 days)

Adapt Atenia's `build_transformer_block` to the exact Llama 2
topology under the HuggingFace naming convention. Reuses M4.5-a's
`RoPE` and Atenia's existing `RmsNorm`.

Naming convention to emit (matches HF TinyLlama checkpoint
exactly):

```
model.embed_tokens.weight
model.layers.{N}.self_attn.q_proj.weight
model.layers.{N}.self_attn.k_proj.weight
model.layers.{N}.self_attn.v_proj.weight
model.layers.{N}.self_attn.o_proj.weight
model.layers.{N}.input_layernorm.weight
model.layers.{N}.post_attention_layernorm.weight
model.layers.{N}.mlp.gate_proj.weight
model.layers.{N}.mlp.up_proj.weight
model.layers.{N}.mlp.down_proj.weight
model.norm.weight
lm_head.weight
```

Output: `TinyLlamaHandles { token_input_id, logits_id, param_ids,
param_names }` analogous to `MiniFluxHandles`. The names go
straight into `WeightMapper::from_param_names_and_ids` (M4-c)
without further translation.

### M4.5-c — End-to-end integration (2 to 3 days)

Download TinyLlama-1.1B safetensors locally (~2.2 GB BF16).
Build the graph via M4.5-b, construct a `WeightMapper`, run
`load_into` against the safetensors file. First real exercise
of the M4-d BF16 → F32 decode path with a non-toy file size.

Run forward on a fixed token sequence (10 tokens, hand-picked).
Capture logits.

Memory pressure note: 4.4 GB F32 weights + activations exceed 8
GB VRAM. Atenia's reaction loop (M3-e) should kick in and
migrate to CPU. This is a **stress test** of the reaction loop
that no synthetic test exercised — opportunity to validate the
full pipeline, risk of surfacing latent bugs in the migration
path that only show with model-sized tensors.

### M4.5-d — Numerical validation (2 to 3 days)

A PyTorch script generates reference outputs for the same fixed
10-token input:

- Per-layer intermediates: post-RMSNorm 1, post-attention,
  post-RMSNorm 2, post-FFN.
- Final logits.

Committed as fixture (NPZ or JSON), regenerable from a script in
the repo. The PyTorch fixture is one-shot; CI does not need
PyTorch.

Rust integration test compares Atenia outputs against the fixture
at every layer boundary. Tolerance: **1e-3** (F32 recompute on
the Atenia side vs BF16 native on the PyTorch side; this drift
is expected, not a bug).

If end-to-end mismatch exceeds 1e-3: debug layer-by-layer
backward from the first divergent intermediate. Common failure
modes documented up front so debugging starts in the right place:

- RMSNorm epsilon difference (`1e-5` vs `1e-6` between
  implementations).
- Softmax subtract-max-before-exp (numeric stability variance).
- Attention scaling order of operations (scale before vs after
  matmul).
- BF16 precision artifacts accumulating across layers (PyTorch
  computes in BF16 throughout; Atenia computes in F32 after the
  load downcast — the reference run should be in F32 as well to
  match, or an explicit drift budget given to each layer).

## Risks

1. **RoPE layout confusion (M4.5-a)** — half-split vs interleaved
   produces silently wrong outputs that look like model bugs.
   Mitigation: Test 1 (fixture canary) catches it immediately.
   High probability of hit during initial development; cheap to
   detect; cheap to fix (transpose two indices).

2. **Numerical drift debugging (M4.5-d)** — the headline risk.
   Layer-by-layer investigation can take days. Mitigation: validate
   each op (RMSNorm, softmax, attention) against PyTorch in
   isolation before the end-to-end run. If standalone op tests pass
   but end-to-end drifts, the bug is in composition (attention
   masking, residual, layer ordering), which narrows the search
   space.

3. **Memory pressure with TinyLlama F32 4.4 GB (M4.5-c)** — exceeds
   VRAM, fits in RAM. Engages the M3-e reaction loop on a tensor
   workload it has never seen. Opportunity (real validation) and
   risk (latent bugs surface here) are the same event. If the
   loop misbehaves, fix is in M3-e territory and unrelated to M4.5
   logic — flag as such and decide whether to fix in-place or
   carve as separate sub-phase.

4. **Scope creep on failure** — if TinyLlama does not work cleanly,
   temptation to switch to another model (Llama 3, gpt2, …) instead
   of debugging. Resist. Each pivot resets the debugging context;
   the right answer is to keep debugging the chosen target until
   it works or the failure is provably architectural (in which
   case re-scope explicitly, not via target swap).

## Out of scope for M4.5

- **Llama 3 / Llama 3.1 / Llama 3.2** — requires GQA + llama3
  scaling RoPE. Both are incremental work on top of M4.5; defer
  until M4.5 lands.
- **Tokenizer integration** — M4.5 works with pre-generated
  integer token tensors. HuggingFace `tokenizers` crate
  integration is M5+.
- **KV cache** — required for efficient autoregressive
  generation, not for single-forward validation. M5+.
- **Token-by-token generation** — needs KV cache + sampling
  strategy. M5+.
- **Training with the loaded model** — M4.5 is forward only.
  Training requires backward through every op of the new graph
  builder, which the existing M3 backward path covers in
  principle but has never been exercised at this scale. M5+.
- **LayerNorm** — only Llama uses RMSNorm. If a future target
  requires LayerNorm (gpt2, BERT, ViT), implement then.
- **Native F16 / BF16 storage** — M4 downcasts on load; native
  storage is a kernels-level milestone (M5 or M6 candidate).
- **Multi-file / sharded safetensors** — TinyLlama is single-file.
  Larger Llamas (7B+) ship sharded; loader extension when needed.

Pre-flight before writing code:

1. Confirm the three pending decisions from the investigation
   report (target TinyLlama, `base_freq: u32`, single commit vs
   sub-divided) — recorded as confirmed in this document.
2. Generate the PyTorch fixture first (script + JSON), so Test 1
   has its reference data ready before the Rust implementation
   starts. This keeps the canary test honest: implementation is
   measured against an externally-produced expectation, not
   against itself.
3. Implement forward + Test 1 first (canary). If Test 1 passes,
   the layout is correct and the rest of the implementation
   proceeds with confidence. If it fails, fix layout before
   writing backward.
